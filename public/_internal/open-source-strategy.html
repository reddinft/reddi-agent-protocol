Component-by-Component Decision
✅ Open Source · Audit Required

Anchor Programs

programs/escrow/

Registry, escrow lock/release, reputation commit/reveal, attestation, dispute, PER privacy release, agent deregister.

Open = trust. Auditable = credible. This IS the protocol. Judges, investors, and users need to read it. Closed smart contracts are a red flag in this space. Needs a proper Adevar Labs (or similar) security audit before mainnet.
✅ Open Source SDK

x402-solana Package

packages/x402-solana/

Payment header parsing, middleware, nonce replay protection, Jupiter auto-swap client.

This is an ecosystem contribution, not the product. Publishing this on npm as @reddi/x402-solana gives us credibility, developer adoption, and hackathon points. The more people use this SDK, the larger our protocol network grows.
✅ Open Source Plugins

Ecosystem Plugins

packages/eliza-plugin-x402/ · packages/sendai-x402/

ElizaOS and Sendai integrations wiring x402 payments into popular agent frameworks.

Distribution channels. Open-sourcing plugins for the top 2–3 agent frameworks means Reddi protocol gets discovered by every ElizaOS and Sendai developer. Network effect > moat here.
✅ Open Source

PER Client

packages/per-client/

Privacy-Enabling Relay client — privacy-preserving payment path for sensitive specialist invocations.

The PER mechanism only works if the community trusts it. Open source is the only way to establish that trust. The on-chain program (release_escrow_per.rs) is already open — the client needs to match.
✅ Open Standard

Capability Taxonomy

lib/capabilities/taxonomy.ts

Task types, input/output modes, privacy modes, runtime capability schema.

This is a standard, not a product. The more agents and orchestrators adopt our taxonomy, the more our registry becomes the canonical agent directory. Open standards compound; closed ones fragment.
✅ Open Source

MCP Tools & Context Contracts

lib/mcp/ · docs/context-contracts/

invoke_specialist, resolve_specialist, heartbeat tool schema; runtime capability + context boundary specs.

Tool schemas need to be public for AI agent frameworks to integrate. This is the API surface of our protocol — keeping it closed prevents the ecosystem from building on it.
✅ Open Source

Web App + Design System

app/ · components/

Next.js pages, DM Sans + Fraunces design system, atomic UI components, marketplace, planner, register flows.

The UI is a demo vehicle, not the moat. Hackathon judges need to see it. The design language is reproducible. Open-sourcing it actually helps — developers fork it to build their own specialist UIs, which grows the ecosystem.
🔒 Close Source · Core Moat

Specialist Ranking Algorithm

lib/onboarding/specialist-index.ts → computeSpecialistRankingScore()

Weighted scoring across feedback count, avg score, attestation agreement ratio, health streak, reputation score, price — determines which specialist wins every dispatch.

This IS the product's intelligence. The ranking algorithm is how Reddi differentiates from a dumb registry. It's how trust compounds over time and how quality rises to the top. Open-sourcing it lets any competitor clone the dispatch logic in hours. Keep closed.
Like Uber's driver dispatch algorithm or Google's PageRank — the ranking logic IS the defensible asset. The parameters, weights, and signal definitions are the secret sauce.
🔒 Close Source · Moat

Orchestration Policy Engine

lib/orchestrator/policy.ts · app/api/planner/tools/invoke/route.ts

Budget enforcement, daily spend tracking, privacy mode routing, attestation gates, fallback logic, maxPerCallUsd enforcement.

The policy engine is what enterprise customers will pay for. Multi-tenant budget controls, per-task cost caps, attestation gates — this is the "management plane" of the protocol. Competitors can build on our open contracts but can't replicate this runtime intelligence without significant engineering. Keep closed, or offer as a paid managed service.
⚠️ Open interface · Close signals

Registry Bridge + Routing Signals

lib/registry/bridge.ts · data/onboarding/specialist-index.json

Merges on-chain + off-chain data. The routing signals (feedbackCount, avgFeedbackScore, attestationAgreements, health_streak) accumulate over time into a data moat.

Open the interface (SpecialistListing type, merge logic) so external tools can build on it. But the routing signals data model and the way signals are weighted into ranking is the growing data asset. As the network scales, the data moat grows larger — protect the signal schema.
✅ Open Source

API Routes — Onboarding

app/api/onboarding/ · app/api/register/

Attestation, healthcheck, capabilities, endpoint probe, runtime check, local Ollama detection.

These are the onboarding rails — anyone building a specialist node needs to understand how to pass these checks. Open = easier adoption, faster ecosystem growth.
⚠️ Review before publishing

Walrus Storage Integration

lib/walrus.ts

Decentralised storage integration — likely for attestation records or audit trail.

Review what data goes to Walrus and whether any of it reveals routing signal data or proprietary schema. If it's just audit logs, open it. If attestation data reveals ranking signals, sanitise first.

What to Do Now — Hackathon Phase

For the hackathon submission (keep everything public)

Judges need to see all the code. For now, the entire repo stays public. The strategic split happens after the hackathon when you're protecting production IP. Focus on:

  1. Extract programs/escrow/ + packages/x402-solana/ into a standalone README with audit instructions — makes it easy for Adevar Labs to scope the audit bounty submission
  2. Add a SECURITY.md with the threat model (escrow attack vectors, replay protection, PER privacy guarantees) — needed for Adevar submission
  3. Add a packages/x402-solana/README.md calling it a standalone SDK — makes the ecosystem contribution obvious to judges and RPC Fast
  4. Add ARCHITECTURE.md at repo root showing the open/closed layer diagram — transparency signal for investors
The Core Moat — What Competitors Cannot Clone

1. The Ranking Algorithm + Accumulated Signals

Every specialist invocation generates routing signals — feedback scores, attestation agreement rates, health streaks. These signals compound into a data moat over time. A competitor can fork the contracts; they cannot fork 6 months of real-world quality signals.

Analogy: GitHub's contribution graph — the data IS the moat, not the rendering.

2. The Policy Engine — Trust-Gated Dispatching

Budget caps, attestation gates, privacy-mode routing, fallback chains — this is enterprise-grade orchestration logic. It took weeks to build and requires deep domain expertise to get right. Open-sourcing it gives competitors a free head start on the hardest part.

Analogy: Cloudflare's WAF ruleset — the infrastructure is open but the rules engine is proprietary.

3. The Specialist Registry Network Effect

The value of the protocol grows with the number of registered, attested, and health-passing specialists. This is a network effect moat — not code. Every new specialist makes the registry more valuable, which attracts more orchestrators, which attracts more specialists.

Analogy: Stripe's merchant network — the rails are open; the trust network is the barrier to entry.

🎯 Strategic Summary

Open what enables adoption. Close what powers quality.

The Anchor programs and x402 SDK are the protocol rails — they must be open, audited, and trusted. The ranking algorithm and policy engine are the intelligence layer — they are the product that enterprises will pay for. The more developers build on the open rails, the more valuable the closed intelligence layer becomes.
