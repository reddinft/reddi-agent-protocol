# RAP MCP Bridge — Live Devnet MCP Plan (2026-05-08)

## Context

PR #254 establishes the safe baseline:

- dry-run MCP bridge
- exactly four non-mutating/safe tools
- Surfpool local payment proof helper
- bounded devnet payment proof helper
- no MCP payment/invoke tools
- no mainnet path

Nissan clarified that devnet funding is approved and a funded devnet wallet is available for top-ups, so the next PR can add live devnet MCP payment semantics if it remains hard-capped and explicitly opt-in.

## Non-negotiable boundary

This PR must not add mainnet support.

This PR must not invoke specialist HTTP endpoints.

This PR must not make devnet mutation the default mode.

This PR must not expose arbitrary transfer tools. It only executes the specific RAP MCP quote/payment semantics for synthetic devnet quotes generated by the bridge.

## Proposed mode gate

Live tools are registered only when all are true:

```bash
REDDI_RAP_MCP_MODE=devnet
RAP_MCP_DEVNET_PROOF_APPROVED=1
RAP_MCP_DEVNET_FUNDER_KEYPAIR=/path/to/devnet/keypair.json
```

Default remains `dry_run`.

If any gate is missing, the server exposes only the four PR #254 tools.

## Proposed tools

### `reddi.prepare_devnet_payment`

Non-mutating readiness tool.

Inputs:

- `quoteId`
- `idempotencyKey`
- optional `maxTotalDebitLamports`

Outputs:

- quote summary
- deterministic payer/specialist/treasury devnet wallet addresses
- configured funder public key hash/address
- balance/readiness checks
- required top-up plan
- total debit cap
- explicit next-step instructions

### `reddi.execute_devnet_payment`

Mutating devnet-only payment tool.

Inputs:

- `quoteId`
- `idempotencyKey`
- `maxTotalDebitLamports`
- `approvalPhrase`: exact `EXECUTE_DEVNET_RAP_PAYMENT`

Behavior:

- rejects unless mode/env gates are present
- rejects if quote is unknown or not devnet-eligible
- rejects if total debit exceeds cap
- idempotent by key and request body hash
- uses configured funder only to top up deterministic proof payer when needed
- executes downstream specialist transfer and protocol fee transfer on Solana devnet
- writes local receipt/disclosure ledger in bridge store
- never invokes specialist endpoint
- never accepts arbitrary recipient addresses

Outputs:

- devnet tx signatures
- balance deltas
- cap respected flag
- disclosure ledger entry
- verification boundary: `solana-devnet`

### `reddi.verify_devnet_receipt`

Mutating: no.

Inputs:

- `quoteId`
- tx signatures

Behavior:

- reads devnet transactions
- verifies expected signer/source/recipient/amount/category where possible
- returns `verified=true` only for devnet semantics proven by signatures
- never claims mainnet/production settlement

## Tests required

- dry-run default exposes exactly four tools
- devnet mode with missing env still exposes only four tools or rejects live calls
- devnet mode with all gates exposes devnet tools

- verify never claims mainnet
